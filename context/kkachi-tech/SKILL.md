## Tech


### 확정된 스택 (2026-07-24 결정)
- 앱: Expo(React Native, TypeScript). EAS Build로 양대 스토어 빌드·제출. UI는 kkachi-design 스킬의 토큰·컴포넌트 스펙을 따릅니다.
- 백엔드: Supabase — Postgres, Auth(애플·구글 로그인), Edge Functions.
- AI: Claude API. 호출은 서버(Edge Function)에서만 합니다. API 키를 앱에 절대 내리지 않습니다.
- 알림: Expo Push(APNs/FCM).

선정 이유: 1인 개발로 연내 양대 스토어 출시가 목표이므로, TypeScript 단일 언어와 관리형 백엔드로 개발·운영 부담을 최소화합니다. Flutter+Firebase안, 자체 백엔드안과 비교 후 결정했습니다.


### 데이터 방침
- 자체 DB(Supabase Postgres)가 기본 저장소이고, 구글 캘린더와 양방향 동기화합니다.
- 사용자의 현재 습관이 구글 캘린더 중심이므로, 동기화 품질이 곧 제품 신뢰입니다.
- 동기화 대상은 일정뿐입니다. 할 일은 자체 DB 전용이며, 구글 Tasks는 사용하지 않습니다. (2026-07-24 확정)
- 구글 캘린더 연결은 선택입니다(2026-07-24 확정). 미연결 상태가 1급 상태입니다 — 동기화 없이 자체 DB만으로 모든 기능이 완전히 동작해야 합니다.


### AI 에이전트 방향
- Claude tool use로 일정·할 일 도구(등록·조회·검색·수정·삭제, 할 일 완료, 준비물 기록)를 제공합니다.
- 시스템 프롬프트에 까치 브랜드 보이스(kkachi-brand)와 현재 시각·타임존·오늘 일정 컨텍스트를 주입합니다.
- "말하면 바로 실행" 원칙에 따라 도구 실행에 확인 절차를 두지 않되, 되돌리기 경로를 함께 설계합니다.
- 까치의 기억은 2층입니다(2026-07-24 확정, kkachi-product 참조). 대화에서 약속·준비물을 감지해 할 일·일정 속성으로 구조화 저장하는 추출 능력이 v1 에이전트의 핵심 요구입니다. 대화 전체를 장기 기억화하는 자유 기억은 두지 않습니다.
- 사진(시간표)·PDF(강의계획서)에서 일정을 추출하는 멀티모달 입력이 v1 범위입니다. Claude API의 비전·문서 입력을 사용합니다. (2026-07-24)


### 아키텍처 v2 초안 (2026-07-24 — 승인 대기)
v1 초안을 같은 날의 제품 결정들(홈=브리핑, 할 일 일급 개념, 2층 기억, 멀티모달 입력, 구글 선택 연결, 일정 완료 없음)에 맞춰 갱신한 판입니다. 운영자 승인 전까지 이 설계로 구현을 시작하지 않습니다.

데이터 모델
- profiles — 타임존, 브리핑 시각, 호칭, 화면 테마, 설정
- events — 일정. 시작·종료 시각, 장소, 준비물 목록, 메모, google_event_id·etag(연결 시). 완료 개념 없음.
- todos — 할 일. 제목, 기한(날짜 필수 아님, 시각은 선택), 완료 시각, 출처(직접 등록 / 대화 약속 포착 / 사진·문서 추출). '오늘·이번 주·이번 달·언젠가' 그룹, '넘어온 n개' 구분, 소멸 규칙(2~3일 이월 시 질문) 트리거는 전부 기한과 오늘 날짜로 계산합니다 — 별도 상태를 저장하지 않습니다.
- conversations·messages — 날짜 단위 대화와 메시지, 첨부 참조. 보존 기간·삭제 정책은 미결.
- attachments — 업로드한 사진·PDF(Supabase Storage 경로).
- google_accounts — 암호화 토큰·sync token. 선택 사항이며, 없어도 전 기능이 동작합니다(미연결 1급).
- quips — 까치의 한마디 풀·이력. 아침 생성 때 변형 몇 개를 미리 뽑아 두고, 탭 회전은 풀에서 소진하며, 일정·할 일이 변하면 재생성합니다. 홈 진입마다 생성하지 않습니다(호출 비용 통제).
- action_log — 에이전트 도구 실행의 전후 스냅샷 기록. 채팅 카드의 '되돌리기'가 이 로그를 근거로 동작합니다.
- usage·subscriptions — 월별 AI 사용량(크레딧식 통합 한도 집행)과 구독 상태(IAP).
- v1 초안 대비 삭제: briefings(발송 기록) — 브리핑은 발송물이 아니라 홈의 상태입니다. reminders는 todos로 대체(리마인더 개념 자체는 기획 미결로 파킹).

Edge Functions
- chat — 대화 + tool use(일정·할 일 CRUD, 준비물 기록), 약속·준비물 포착(2층 기억), 사진·PDF 일정 추출, 되돌리기 실행.
- calendar-sync — 구글 양방향 동기화(연결자만). sync token 증분, etag 충돌은 최신 수정 우선(last-write-wins).
- morning — 사용자별 브리핑 시각에 실행: 한마디 풀 생성 + 아침 푸시 발송.

스케줄(pg_cron) — 분 단위 스윕
1. 브리핑 시각 도달 사용자 → morning 실행
2. 시작 임박 일정 → 직전 리마인드 푸시(장소·준비물 포함)
3. 동기화 폴링(연결자만)

미결 (구현 전 확정 필요)
- Claude 모델 티어(대화·한마디 각각)와 대화당 비용 추정
- 대화 이력 보존·삭제 정책(개인정보처리방침과 연동)
- 구글 OAuth sensitive scope 검증 신청 준비물과 일정 반영
